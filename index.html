Project Details

The importance of IoT devices in our interconnected world cannot be overstated. These devices play a crucial role in collecting and exchanging data, enabling smart systems and enhancing efficiency in various domains. However, due to their resource constraints and limited computing power, IoT devices are inherently more vulnerable to security threats, making data security a paramount concern.


To ensure the integrity, confidentiality, and privacy of data transmitted and stored by IoT devices, the selection and implementation of robust security algorithms are of utmost importance. Cryptographic algorithms provide the necessary tools to safeguard data and communications in IoT environments. These algorithms enable encryption and decryption processes that protect sensitive information from unauthorized access and tampering.


The performance evaluation of cryptographic algorithms is a key objective of our project. By assessing the performance metrics of these algorithms, such as execution time, memory usage, we aim to identify the most efficient solutions for resource-constrained IoT devices. This evaluation will enable us to strike a balance between data security and computational efficiency, ensuring that the chosen algorithms are both robust and suitable for the limitations of IoT devices. Ultimately, our project aims to contribute to the development of secure and efficient IoT systems.

Algorithms & Test Environments


The algorithms we have decided to consider are listed below:


  1. Cipher Block Chaining (CBC): CBC is a symmetric block cipher mode that adds an extra layer of security by chaining the ciphertext blocks together. It XORs each plaintext block with the previous ciphertext block before encryption, ensuring that identical plaintext blocks produce different ciphertext blocks. This makes it resistant to certain types of attacks.

  2. Output Feedback (OFB): OFB is a symmetric stream cipher mode that generates a keystream independent of the plaintext. It encrypts an initialization vector (IV) and uses the resulting ciphertext as the keystream. The keystream is XORed with the plaintext to produce the ciphertext. OFB is known for its error propagation properties and is resistant to bit flipping attacks.

  3. Rivest-Shamir-Adleman (RSA): RSA is an asymmetric encryption algorithm widely used for secure communication and digital signatures. It relies on the difficulty of factoring large prime numbers. RSA uses a public-private key pair, where the public key is used for encryption, and the private key is used for decryption and digital signing.

  4. Trivium: Trivium is a lightweight symmetric stream cipher known for its simplicity and high performance on hardware platforms. It utilizes three linear feedback shift registers (LFSRs) and a non-linear combining function to generate a keystream. The keystream is XORed with the plaintext to produce the ciphertext.

  5. Present: Present is a lightweight asymmetric block cipher designed for resource-constrained environments. It operates on small data blocks and uses a compact hardware implementation. Present offers a good balance between security and efficiency and is suitable for lightweight cryptographic operations.

These algorithms serve different cryptographic purposes and have distinct characteristics that make them suitable for various applications and environments.

Results


In our test results, we observed that RSA exhibited the lowest performance among the tested algorithms, while the lightweight algorithms, specifically Trivium and Present, demonstrated the highest performance. Trivium and Present, which fall under the category of lightweight algorithms, outperformed the others in terms of efficiency and speed. These findings highlight the advantage of using lightweight cryptographic algorithms, such as Trivium and Present, in resource-constrained environments where performance optimization is crucial. By leveraging their optimized design and implementation, Trivium and Present offer superior performance and make them well-suited for applications requiring efficient and secure cryptographic operations.


RSA, when compared to Triple DES OFB, Triple DES CBC, and lightweight algorithms, took longer to execute due to several reasons:


  1. Asymmetric Encryption: RSA is an asymmetric encryption algorithm, which means it uses a public key for encryption and a private key for decryption. Asymmetric encryption algorithms generally perform slower than block cipher algorithms. Consequently, the execution time of RSA can be longer compared to other block cipher algorithms.

  2. Key Length: RSA typically employs longer keys. Longer keys can enhance security, but encryption and decryption operations require more computational power. As the key length increases, the processing time also increases.

  3. Modular Arithmetic Operations: RSA relies on modular arithmetic operations based on number theory. Performing arithmetic operations with large numbers can be more complex and slower.

Lightweight algorithms are faster than Triple DES OFB and CBC due to several reasons.


  1. Simplified Design: Lightweight algorithms are specifically designed to have simpler and more streamlined designs compared to complex algorithms like Triple DES. They prioritize efficiency and speed by eliminating unnecessary operations and reducing computational overhead.

  2. Reduced Key Sizes: Lightweight algorithms often use shorter key sizes compared to Triple DES. While shorter key sizes may result in lower security levels, they contribute to faster encryption and decryption operations. The reduced number of key bits decreases the computational load, leading to improved performance.

  3. Efficient Implementation: Lightweight algorithms are optimized for resource-constrained devices and environments where computational power and memory capacity are limited. They often have efficient and compact implementations that make effective use of available resources. This optimization results in faster execution and lower memory requirements.



When examining the file sizes, it was observed that the Present algorithm exhibited better performance in smaller files, while the Trivium algorithm showed better performance as the file size increased. Additionally, as the file size increased, algorithms other than lightweight algorithms took longer to yield results.


In addition, the Present algorithm, due to its lower memory usage, did not encounter the "out of memory" error as observed in the other algorithms when processing a 100MB file. This advantage can be attributed to several factors that were present in the algorithm's design.
